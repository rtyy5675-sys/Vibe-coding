# v0.1 操作手册

本手册面向执行人员。v0.1 只要求跑通评估链路，不要求接入真实模型。

## 1. 新增 case

1. 选择 case 类型：STT、TTS 或 LLM。
2. 在 `cases/` 对应 JSONL 文件中新增一行。
3. 确认每条 case 至少包含：
   - 唯一 `case_id`
   - 业务 `scenario`
   - `language`
   - 输入内容或音频引用
   - 期望结果或参考文本
   - `critical_entities`
   - `risk_tags`
4. 高风险 case 标记 `must_not_fail` 或 `human_review_required`。
5. 保存后抽查 JSONL：每行必须是完整 JSON。

## 2. 准备 outputs

每次执行在 `runs/` 下建立一个新目录，建议命名：

```text
runs/YYYYMMDD_run_name/
```

无 API key 时：

1. 手动或脚本生成模拟输出。
2. 保留与 case 对应的 `case_id`。
3. 明确标记 `output_mode: mock`。

真实模型模式：

1. 记录模型名称、版本、provider 和运行时间。
2. 保留原始模型输出。
3. 不覆盖旧 run。
4. 明确标记 `output_mode: real_model`。

## 3. 运行 scorer

按对象类型执行评分：

- STT：计算 WER/CER，并检查金额、日期、状态、订单号、否定词等关键实体。
- TTS：先记录 TTS 输出，再做 STT 回转文本检查；自然度进入人工 MOS 评分。
- LLM：检查结构化输出、关键实体、schema、固定断言和失败原因。

每次评分后确认：

- 有总样本数。
- 有通过数和失败数。
- 有待人工复核数。
- 每个失败样本能追溯到 `case_id`。
- Critical 失败被单独标出。

## 4. 更新报告

将评分结果汇总到 `reports/`。

报告至少包含：

- 本次 run 名称和时间。
- 使用模拟输出还是真实模型输出。
- 样本范围：STT / TTS / LLM 各多少条。
- 自动评分指标。
- Critical / Major / Minor 失败数量。
- 待人工复核清单。
- 不能从本次结果推出的结论。

如果本次使用模拟输出，报告必须写明：

```text
本次结果只能证明评估系统可运行，不能证明模型优劣。
```

## 5. 更新看板

更新看板前先确认报告数据已生成。

看板字段建议至少包含：

- run 名称
- output mode
- 模型或模拟输出来源
- 样本数
- 失败数
- Critical 失败数
- 待人工复核数
- 最新更新时间

更新后检查：

1. 看板能正常打开。
2. 三类对象 STT / TTS / LLM 都能显示。
3. 数字与 `reports/` 汇总一致。
4. 模拟输出不会被展示成真实模型评估结果。

## 6. 完成标准

一次 v0.1 执行完成，需要满足：

- case 可读取。
- outputs 可追溯到 case。
- scorer 可生成结果。
- reports 有清晰边界说明。
- dashboard 能展示核心指标。
- 人工复核项没有丢失。
