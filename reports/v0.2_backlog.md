# v0.2 Backlog

v0.2 的目标是从“评估系统可运行”推进到“小样本真实或半真实评估可用于内部评审”。

## 1. 工具接入

- 接入 Promptfoo，用于 LLM prompt、provider、case 的矩阵评估。
- 接入 DeepEval，用于关键业务任务的回归测试。
- 接入 Ragas，用于 RAG faithfulness、context recall、answer relevancy。
- 保留 OpenAI Evals 作为 OpenAI API 栈的可选执行器。
- 统一工具输出到 `reports/tool_results.jsonl`。
- 建立工具输出到看板字段的转换脚本。

## 2. 样本扩展

- STT 样本扩展到至少 30 条。
- TTS 样本扩展到至少 30 条。
- LLM 样本扩展到至少 50 条。
- 每类样本覆盖常见场景、边界场景和高风险场景。
- 为金额、日期、订单号、否定词、状态词建立 Critical case。
- 为多语言或混合语言 case 增加单独标签。

## 3. 真实数据治理

- 明确真实数据是否可用于评估。
- 建立脱敏规则：姓名、手机号、订单号、地址、账户信息。
- 记录每批数据来源、授权状态和处理人员。
- 将合成数据、脱敏数据、真实数据分开标记。
- 对包含敏感信息的 run 设置访问边界。

## 4. 生产监控

- 定义线上抽样策略。
- 定义 STT / TTS / LLM 的线上失败事件结构。
- 建立 Critical 失败报警口径。
- 将线上失败样本回流到 case 库。
- 建立每周或每版本复测节奏。

## 5. 人工评分一致性

- 为 TTS MOS 建立评分说明。
- 为 LLM 开放答案建立人工评分 rubric。
- 至少安排两名评分人员对同一批样本交叉评分。
- 统计评分分歧，并保留仲裁结论。
- 抽查自动评分与人工评分不一致的 case。

## 6. 发布材料

- 形成 v0.2 Review 报告。
- 增加失败样本附录。
- 增加真实或脱敏样本的数据声明。
- 增加候选模型配置表。
- 增加人工复核摘要。
